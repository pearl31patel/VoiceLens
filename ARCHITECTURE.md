# Architecture

The system is a small Python QA harness for voice agents. `run_calls.py` creates outbound calls through Twilio and always targets the assessment number. When the call is answered, Twilio requests `/twiml`, and the FastAPI app returns TwiML that connects the live call audio to `/media` using a bidirectional Media Stream. The WebSocket receives the agent's audio, forwards it to Deepgram for real-time transcription, and sends synthesized patient audio back to Twilio as 8 kHz mu-law frames so the assessment agent hears it like a normal caller.

The conversation logic is scenario-driven rather than fully scripted. Each scenario defines the patient identity, intent, goal, and edge case. The `PatientBrain` uses the latest agent transcript plus the scenario goal to choose the next short, natural patient response. This keeps the bot coherent while still allowing the tested AI agent to take the conversation in unexpected directions. I chose this design because the challenge rewards realistic patient interaction, sensible turn-taking, and active steering toward test outcomes, not a rigid benchmark script. After the calls, transcripts and recordings are saved as evidence, and `analyzer.py` creates a bug report by reviewing the actual call transcripts.
